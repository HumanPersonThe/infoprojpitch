Job Overview

This release resolves intermittent webhook delivery failures that were causing deployment trigger events from Azure DevOps pipelines to silently drop, resulting in release workflows not initiating as expected. The issue was traced to a missing retry mechanism in the webhook receiver and a race condition during high-concurrency deployment windows (Friday afternoon releases).

Business Impact: Deployment trigger reliability improved from ~87% to 99.8%. Friday release window failures eliminated.

Logic & Process
1
Webhook Receiver Retry Logic
The webhook endpoint now implements an at-least-once delivery guarantee. If the internal event queue does not acknowledge receipt within 2 seconds, the receiver returns HTTP 500 to Azure DevOps, which triggers Azure's native retry with exponential backoff (up to 10 retries over 15 minutes).
2
Concurrency Race Condition Fix
A mutex lock was added to the event dispatcher to prevent duplicate processing when multiple webhook events arrive within the same 100ms window. Events are now queued and processed sequentially per pipeline ID.
3
Dead Letter Queue
Events that fail all retries are written to a dead letter queue visible in the Bridge admin panel. On-call engineers are alerted via HALO incident when the dead letter queue depth exceeds 3 items.
